# CURSOR RULES — androidchity.ru

> **Читать первым при каждой новой сессии или потере контекста.**

---

## 0. Контекст проекта (не терять)

Это **статический HTML-сайт**, восстановленный/перенесённый с WordPress на статические файлы.

- **Не WordPress, не PHP, не шаблоны WP.**
- Сейчас — самописный HTML/CSS/JS сайт.
- Деплой: GitHub Pages / Plesk (статика).
- Локальный запуск: `node server.js` (порт 8080).

**Главная цель проекта:**
- Сохранить визуал и функциональность оригинального сайта.
- Сделать сайт быстрым (минимум лишнего JS/CSS).
- Использовать как платформу для SEO-страниц под запросы:
  - Скачать Android-приложения / APK
  - Букмекерские конторы / беттинг (Android betting pages)
  - Мультиязычный контент (в т.ч. арабские страницы)

---

## 1. Главное правило: НЕ ЛОМАТЬ САЙТ

**Запрещено:**
- Массово переписывать структуру файлов без предварительного анализа.
- Удалять JS/CSS «потому что кажется лишним» без проверки использования.
- Менять классы/селекторы без проверки их использования в JS.
- Копировать HTML из DevTools вместе с `.owl-item`, `.owl-stage`, `.cloned`, inline `transform`.
- Вносить изменения сразу в несколько независимых блоков, если задача только про один.

**Разрешено:**
- Точечные изменения после чтения актуального состояния файлов.
- Backup/копия перед рискованными правками.
- Добавление комментариев в код для поддержки.

## 2. Обязательный алгоритм перед КАЖДОЙ задачей

### Шаг 1. Прочитать контекст проекта
Найди и прочитай:
- `CURSOR_RULES.md` (этот файл) — **всегда первым**
- `PROJECT_MAP.md` — карта проекта
- `CHANGELOG_DEV.md` — что менялось недавно
- Файлы, относящиеся к задаче (HTML/CSS/JS)

Если контекст утерян (новая сессия) — пересобери понимание из файлов, опиши какие файлы отвечают за нужный блок.

### Шаг 2. Проверить последние изменения

```bash
git status
git diff
git log --oneline -n 10
```

Цель: понять, что было изменено, и не затереть это.

### Шаг 3. Найти все связанные файлы

Если задача касается, например, слайдера — обязательно найди:
- HTML разметку слайдера
- CSS для этого блока
- JS инициализацию (или её отсутствие)
- Подключение библиотек (owl/swiper/jquery и т.п.)

Нельзя править «вслепую» только HTML.

### Шаг 4. Сделать безопасную точку

Перед рискованными изменениями:
- Сохранить diff или сделать коммит.
- Если изменения масштабные — создать отдельную ветку.

## 3. Технические правила проекта

### 3.1 Проект — статический HTML
- Нельзя добавлять WP-функции (`wp_head`, `the_content`, `wp_enqueue_script`).
- Нельзя делать зависимость от БД.
- Нельзя писать код, работающий только в WP.

### 3.2 Сохранять URL-структуру страниц
Структура файлов должна соответствовать URL (например `page/2/index.html`). Ссылки не должны ломаться.

### 3.3 Сохранять визуал
Любое изменение должно:
- либо исправлять баг,
- либо улучшать скорость,
- но **не менять внешний вид без запроса**.

### 3.4 Приоритет скорости
- Меньше лишнего JS.
- Меньше лишнего CSS.
- Правильная загрузка изображений (lazy, width/height атрибуты).
- Отсутствие дублирующихся библиотек.

## 4. Правила работы со слайдерами

### 4.1 Нельзя копировать DOM из инспектора браузера
Запрещено переносить в HTML: `.owl-stage`, `.owl-item`, `.cloned`, inline `style="transform: translate3d(...)"`.
Это результат работы JS, а не исходный шаблон.

### 4.2 Правильный подход
Слайдер должен состоять из:
- Контейнера (например `.owl-carousel`, `.owl-mobile`, `.slider`)
- Исходных слайдов (например `.mobile-slide`)
- JS-инициализации в основном скрипте проекта

### 4.3 Если слайдер не работает — проверяй по шагам:
1. Есть ли HTML-элементы слайдов в DOM?
2. Подключены ли CSS/JS библиотеки слайдера?
3. Загружается ли библиотека без ошибок?
4. Вызывается ли инициализация после `DOMContentLoaded`?
5. Нет ли ошибок в консоли?
6. Нет ли конфликта из-за дублей jQuery/библиотеки?

### 4.4 Если библиотеку восстановить не удалось
Сделать лёгкий современный слайдер (без jQuery) и сохранить внешний вид: dots, autoplay, ручное переключение, swipe на мобилке.

## 5. Правила HTML/CSS/JS

### 5.1 HTML
- Не удалять важные `class`/`id` без проверки в JS/CSS.
- Сохранять `alt` у изображений.
- Не ломать арабскую верстку (`dir="rtl"`, RTL-логика).

### 5.2 CSS
- Не вносить глобальные стили при локальной задаче.
- Не использовать слишком общие селекторы (`img`, `div`, `p`) для точечных правок.
- При исправлении пустого пространства — сначала найти источник (`margin`/`padding`/`min-height`/`flex`/`gap`).

### 5.3 JS
- Новые скрипты подключать с `defer`.
- Новую логику писать в один основной файл (или модульно).
- Не дублировать обработчики на один элемент.
- Инициализация должна быть идемпотентной (повторный запуск не ломает блок).

## 7. Формат работы над задачей

**Шаг A. Анализ**
- Что сломано / что нужно сделать
- Какие файлы участвуют
- Почему это происходит (гипотеза)

**Шаг B. План**
- Какие конкретно изменения будут внесены
- Что проверим после

**Шаг C. Реализация**
- Точечные изменения
- Не трогать несвязанные блоки

**Шаг D. Проверка**
- Страница открывается без ошибок
- Нет критичных ошибок JS в консоли
- Стили не «поехали»
- Слайдер (если трогали): autoplay, dots, ручное переключение, swipe на мобилке
- Нет 404 на изображения/CSS/JS
- Изменения не затронули другие страницы (проверить 2–3)

**Шаг E. Отчёт**
- Какие файлы изменены
- Что конкретно сделано
- Что проверить вручную
- Есть ли риски / что осталось

## 8. СТОП-правила

Остановись и сообщи (не ломай дальше), если:
- Не можешь найти исходный HTML блока.
- На странице 2+ одинаковых слайдера с разной логикой.
- Библиотека подключена, но инициализация спрятана в минифицированном общем файле.
- После изменения появился сдвиг верстки в других блоках.
- Изменения затрагивают больше 3 файлов, а задача маленькая.

В этом случае: перечисли найденные файлы, предложи безопасный вариант.

## 9. SEO и бизнес-правила

Проект используется для SEO/контентных страниц под Android и беттинг-тематику.

- **Нельзя** ломать тексты, заголовки, таблицы, CTA-кнопки.
- **Нельзя** удалять контентные блоки без запроса.
- **Нужно** сохранять контент на страницах: `/download/`, `/register/`, `/cheats/`, `/cheatz/`, юридических страницах.

## 10. Git-правила

Коммит с понятным названием:
```
fix: restore homepage slider autoplay and dots
fix: remove excessive whitespace in content block
perf: optimize image loading on apk page
feat: add new betting landing page
```

Не смешивать в один коммит несвязанные изменения.
